# Phase 1: Evaluation Charts for Prompt Evolution

**Simplified experimental plan — 3 conditions, 5 charts.**

---

## Conditions

| Label | Description | Student | Teacher |
|-------|-------------|---------|---------|
| **B** (Baseline) | Default tau2-bench prompt, no modifications | Qwen 3.5-35B-A3B | — |
| **K** (Kimi-evolved) | Prompt evolved via Kimi K2.5 reflection loop | Qwen 3.5-35B-A3B | Kimi K2.5 |
| **F** (Frontier ceiling) | Use the teacher model directly as the agent | Kimi K2.5 | — |

Each condition is evaluated independently per domain (airline, retail, telecom).
No cross-domain transfer — each domain has its own prompt and tool set.

---

## Charts

### 1. Pass Rate Over Iterations

- **Type**: Line chart
- **X-axis**: Iteration (0 = baseline, 1, 2, 3, ...)
- **Y-axis**: pass^1 (fraction of tasks passed)
- **Lines**: One per domain (airline, retail, telecom)
- **Purpose**: Shows the evolution curve — whether pass rate improves with each iteration and where it saturates.

### 2. Baseline vs. Evolved vs. Frontier (Bar Chart)

- **Type**: Grouped bar chart
- **Groups**: One group per domain
- **Bars**: B (baseline), K (best evolved iteration), F (frontier ceiling)
- **Y-axis**: pass^1
- **Purpose**: Direct comparison of the three conditions. The key visual is how close K gets to F.

### 3. Gap Closure

- **Type**: Single bar per domain
- **Y-axis**: % of the (B → F) gap that K closes, i.e. `(K - B) / (F - B) * 100`
- **Purpose**: Normalizes for domain difficulty. A 50% gap closure means the evolved prompt captured half the teacher model's advantage through prompt patching alone.

### 4. Failure Type Distribution (Before / After)

- **Type**: Stacked bar chart
- **Groups**: One pair (before / after) per domain
- **Stacks**: TOOL_MISUSE, POLICY_VIOLATION, REASONING_ERROR, COMMUNICATION_ERROR
- **Y-axis**: Count of failures
- **Purpose**: Shows which failure types the evolution framework resolves and which persist.

### 5. Per-Task Pass/Fail Flip Matrix

- **Type**: Binary heatmap per domain
- **Rows**: Task IDs
- **Columns**: Iterations (0, 1, 2, ...)
- **Colors**: Green = pass (reward = 1.0), Red = fail (reward < 1.0)
- **Purpose**: Reveals whether patches cause regressions (green → red) or progressive fixes (red → green). Also shows which tasks are consistently hard.

---

## Data Requirements

Each chart pulls from the results JSON files produced by `evo.evaluation.runner.run_baseline()`.

Key fields per simulation:
- `task_id` — maps to heatmap rows
- `reward_info.reward` — 1.0 = pass, <1.0 = fail
- `reward_info.reward_breakdown` — per-criterion scores

Failure type classification comes from the teacher model's diagnosis (stored in patch metadata).

---

## Implementation

Charts are generated by `evo/analysis/charts.py`, which:
1. Loads results JSON files from `results/`
2. Extracts pass/fail per task per iteration
3. Renders all 5 charts to `results/figures/`

Run via: `uv run python -m evo.analysis.charts`
